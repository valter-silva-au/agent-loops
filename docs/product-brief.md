# Product Brief: Agent Loop Framework

**Date:** 2026-03-10
**Author:** analyst

---

## Vision

A Python framework that takes a structured product spec (`prd.json`) and autonomously builds, tests, and commits working software overnight using AI agent loops. The human writes the spec and reviews the output; the machine handles everything in between. This becomes the foundation for a fully autonomous SaaS factory.

## Problem Statement

Building software with AI assistants today is interactive -- the human must stay in the loop, approving each step, managing context, and restarting sessions when they degrade. This limits AI-assisted development to the hours a human is actively at the keyboard. The potential of AI to work autonomously around the clock is wasted.

### Current State

- **Karpathy's autoresearch** demonstrated continuous AI loops for ML training but is domain-specific (neural network experiments, validation BPB metric). Not applicable to general software engineering.
- **Claude Code CLI** can be run in bash loops manually, but without structured state management, safety guardrails, or progress tracking, it's fragile and risky (runaway API costs, repeated failures, context degradation).
- **AI coding tools** (Cursor, Aider, Copilot) are interactive by design. They augment a human developer but cannot operate unattended.
- The human remains the bottleneck: writing code, reviewing diffs, restarting sessions, managing git manually.

### Desired State

- A developer writes a `prd.json` spec in the evening and runs `agent-loops run --prd spec.json`
- The framework spawns fresh Claude Code agents in a loop, each picking up where the last left off via externalized state (git commits, `progress.txt`, `prd.json` task status)
- Safety guardrails (token budgets, iteration caps, gutter detection) prevent runaway costs and failures
- By morning, there is a tested, committed codebase implementing the spec -- ready for human review and deployment
- The loop is observable: a clear log of what each agent iteration attempted, succeeded, or failed at

## Key Features

1. **Ralph Wiggum Loop Harness**: An infinite loop that spawns fresh Claude Code / Agent SDK sessions, feeds them the current project state, and restarts on completion or failure. Externalized memory via git commits, `progress.txt`, and `prd.json` task tracking.

2. **Safety Guardrails ("Durable Box")**: Token budget caps with hard kills at threshold, maximum iteration limits, gutter detection (identifies when an agent is thrashing on the same failure), and idempotency checks to prevent duplicate side-effects (double commits, repeated file overwrites).

3. **Structured Spec Input (`prd.json`)**: A machine-readable product spec format that defines tasks, acceptance criteria, and completion status. Each agent reads it to decide what to work on next; updates it to mark tasks done.

4. **Observable Progress**: A `progress.txt` log and git commit history that provide a clear audit trail. Each iteration records: what task was attempted, what was changed, whether tests passed, and what the agent learned.

5. **CLI Interface**: A Python CLI (`agent-loops run`) with configuration for budget limits, iteration caps, target directory, and spec file path.

## Constraints

- **Budget-conscious**: API token costs must be capped and tracked. A single overnight run should cost < $50 for a typical small SaaS. Hard budget kills are mandatory.
- **PRIS Act compliance (July 2026)**: Any SaaS products generated by the framework must embed privacy-by-design patterns. The framework itself must not log or persist personal data unnecessarily.
- **Python ecosystem**: Must be built in Python to align with Claude Agent SDK and the autoresearch lineage.
- **Single-agent MVP**: The first version uses a single agent per loop iteration. Multi-agent swarms (parallel workers, judge agents) are deferred.
- **Claude-specific**: The MVP targets Anthropic's Claude (Claude Code CLI and/or Agent SDK). Model-agnostic support is a future concern.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Runaway API costs from infinite loops | High | High | Hard token budget caps with process kill at threshold; iteration limits |
| Agent thrashing (repeating same failed action) | High | Medium | Gutter detection: if same error appears N times consecutively, pause and log |
| Context degradation producing broken code | Medium | High | Fresh agent per iteration; state externalized to files, not agent memory |
| Generated code has security vulnerabilities | Medium | High | Mandate test suites include security linting; human review before deploy |
| Claude API rate limits or outages halt the loop | Medium | Low | Exponential backoff with jitter; graceful pause and resume |
| prd.json format too rigid / not expressive enough | Medium | Medium | Design the format iteratively; start simple, extend as real usage reveals gaps |

## Open Questions

- Should the framework invoke Claude Code CLI as a subprocess, or use the Claude Agent SDK directly as a library? (Trade-off: CLI is simpler but less controllable; SDK gives hooks but more code to write)
- What is the optimal granularity for tasks in `prd.json`? Too coarse and the agent can't finish in one iteration; too fine and overhead dominates.
- How should the framework handle merge conflicts or dirty git state when it starts?
- Should `progress.txt` be structured (JSON/YAML) or freeform markdown that the agent writes naturally?
- What test runner should the framework mandate for generated projects? Or should it be configurable per project?
